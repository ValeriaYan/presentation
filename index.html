YAML (YAML Ain't a Markup Language) is a human—friendly data serialization standard for all programming languages.

The main objectives of the creation:
  1. be understandable to a person;
  2. support data structures native to programming languages;
  3. be portable between programming languages;
  4. support streaming processing;
  5. be expressive and expandable;
  6. be easy to implement and use
Advantages of YAML
  • Convenience of perception and mastering by a person
  • Independence from the programming language
  • Uniqueness
  • Suitability for version control
  • Syntax strictness
  • Processing speed
    • Examples of using YAML
    • configuration files
    • data sharing by programs in different languages
    • debugging complex data structures
    • messaging between processes
    • data storage
    YAML Features
    Multi document support

    Documents are separated by three hyphens (—):

    							
---
player: playerOne
action: attack (miss)
---
player: playerTwo
action: attack (hit)
---
    Support for comments

    YAML allows you to add comments after the character #

    							
key: 
#This is a one-line comment
   	- value line 5
#This is a 
#multiline comment
	- value line 13
    Easy-to-read syntax

    The syntax of YAML files uses an indentation system. It is necessary to use spaces rather than tabs to avoid confusion.

    							
#YAML
 Imaro:
 author: Charles R. Saunders
 language: English
 publication-year: 1981
 pages: 224
							
						
    							
{
	"Imaro": {
		"author": "Charles R. Saunders",
		"language": "English",
		"publication-year": "1981",
		"pages": 224
	}
}
    Explicit and implicit typing

    To use a specific type, you need to write !![type] before the value.

    							
# This value is converted to int:
is-an-int: !!int 14.10
# Turns any value into a string:
is-a-str: !!str 67.43
# The value must be boolean:
is-a-bool: !!bool yes
    Anchors or variables

    Anchors allow you to avoid repeating configuration fragments.

    In JSON we would have to duplicate these configs

     
							
	{
	"production": {
		"node_version": "13.0.0",
		"os": "ubuntu",
		"package_manager": "yarn",
		"run": ["yarn install", "NODE_ENV=${ENVIRONMENT} yarn build"],
		"env": {
			"ENVIRONMENT": "production"
		}
	},
	"development": {
		"node_version": "13.0.0",
		"os": "ubuntu",
		"package_manager": "yarn",
		"run": ["yarn install", "NODE_ENV=${ENVIRONMENT} yarn build"],
		"env": {
			"ENVIRONMENT": "development"
		}
	}
	}

    In YAML, this problem is solved by variables. Variables are denoted by an ampersand (&), and their values are substituted using an asterisk (*)

    							
base-config: &base
  node_version: 13.0.0
  os: ubuntu
  package_manager: yarn
  run:
    - yarn install
    - NODE_ENV=${ENVIRONMENT} yarn build
production:
	# Embedding all "base" attributes and values in production
	<<: *base
	env:
	  - ENVIRONMENT: production
staging:
	# Embedding all "base" attributes and values in staging
	<<: *base
	env:
	  - ENVIRONMENT: staging
    Other YAML features
    YAML vs JSON
    YAML
    • YAML allows you to describe real-world objects in a readable format.
    • Provides more features than JSON for creating complex data structures. This is a strict superset of JSON, and therefore all valid JSON documents are also valid YAML documents.
    • YAML supports in-line processing
    • YAML has the smallest size in serialized form.
    JSON
    • JSON is a format for storing and transmitting data.
    • Comments in JSON are not supported.
    • Surpasses YAML in parsing and serialization speed.
    • This format is most often used in web development, for example, to transfer data from a server to a web page.
    What is the best way to use it?

    If you just need to store and transfer data, or if you have a fairly simple configuration in a couple of lines, then there is no point in complicating and using additional YAML features, it is better to choose JSON. And if you have a rather large and ornate file, then it is definitely worth considering YAML.
